# Recommendation Readiness Review

WP-06A — Batch 12: Recommendation Intelligence Engine

## 1. What Was Built

The platform's core deterministic decision engine (`Apps/ai-backend/src/modules/recommendation/`), structurally mirroring the certified `modules/goal`, `modules/roadmap`, and `modules/assessment` modules: aggregate + entities + value objects + invariants + domain events, a pure-arithmetic `RecommendationEngine` implementing all 8 engine responsibilities and all 7 scoring-model values, application commands/queries/services, Mongo persistence, an Outbox-based event publisher, a distributed lock, RBAC-guarded REST API, and observability (structured logs + 5 Prometheus metrics). Full detail in `RecommendationImplementationReport.md`; item-by-item compliance in `RecommendationCertificationChecklist.md`.

322/322 tests pass across the whole `ai-backend` app (54 suites, no regressions), including 71 new Recommendation-specific tests — the largest single-batch test addition of the four DDD modules built so far (Goal/Roadmap/Assessment/Recommendation), reflecting the size of the scoring model and the 8-way strategy decision tree.

## 2. Genuine Strengths

- **Every one of the 8 learning strategies is independently reachable and tested**, with an explicit, ordered decision tree (§4 of the implementation report) rather than an implicit if/else chain a reader would have to reverse-engineer. This directly satisfies the spec's "no hidden decision logic" constraint in a way that's checkable, not just claimed.
- **Explainability is structural, not decorative** — `RecommendationReason` (summary + evidence array) is a required constructor argument on every `RecommendationItem`, so it is impossible to construct an item without it; a dedicated test asserts every item in every generated computation has non-empty reason and evidence.
- **Dates are reproducible by construction** — the engine takes an explicit `referenceDate` input rather than calling `Date.now()` internally, so `ReviewSchedule.dueDate` participates in the determinism guarantee instead of being an exception to it. This is a stricter standard than was strictly necessary and closes a category of bug the other modules didn't have to deal with (Roadmap/Assessment don't compute future calendar dates from "now").
- **No new cross-module coupling**, continuing the precedent from Roadmap (`goalSnapshot`) and Assessment (caller-supplied signals): `GenerateRecommendationsCommand` takes every signal it needs as explicit input, so the Recommendation module cannot silently drift out of sync with Goal/Roadmap/Assessment internals since it never touches them at runtime.

## 3. Deliberate Scope Deviations From the Spec (flagged, not silently made)

1. **`GetRecommendationHistory` HTTP endpoint added, not listed in the spec's API section.** Same recurring gap class as the Roadmap and Assessment batches: the query is required, the URL isn't given. Added `GET /recommendation/:id/history`.
2. **`LearningStrategyChanged` is emitted on every generation, not only when a strategy actually changes.** Because `GenerateRecommendations` creates a brand-new `Recommendation` aggregate each time (there is no "regenerate an existing recommendation" command in this spec, unlike Roadmap's `RegenerateRoadmap`), there is no previous state *within this aggregate* to diff against. The event currently means "these are this generation's strategy assignments," not "the strategy for skill X changed from what it was last time." If the intended semantics were the latter, it would require querying prior `Recommendation` aggregates for the same `roadmapId`/`learnerId` — a cross-aggregate read this module deliberately does not do, to avoid a second axis of coupling beyond the input-snapshot boundary. This is an inferred reading of an ambiguous event name and should be confirmed.
3. **Approve/Reject act on the whole `Recommendation`, not per-item.** The spec's `ApproveRecommendation`/`RejectRecommendation` commands take only a `recommendationId`, not an item id, so this implementation accepts or dismisses the entire generated set atomically. If the intended product experience is "approve the task-priority suggestions but reject the difficulty adjustment," this batch does not support that — a learner or teacher can only accept-all or reject-all for a given generation. Re-running `GenerateRecommendations` is the only way to get a different set.
4. **`skillArea` correlation risk is now compounded across two upstream modules.** This was flagged as a single-module data-quality risk in the Assessment readiness review; Recommendation inherits it twice over, since it groups by `skillArea` across both the Roadmap's `tasks[].skillArea` and the Assessment's `competencies[].skillArea`/`knowledgeGaps[].skillArea`. If any of the three producers (Roadmap task labels, Assessment competency labels, Assessment gap labels) drift out of exact-string agreement for the same conceptual skill, the Recommendation engine will silently treat them as unrelated areas rather than erroring — need scores, gap resolution, and strategy selection would all quietly degrade for that skill without any exception being thrown. This is the most consequential structural risk carried into this batch, precisely because Recommendation is the point where all three upstream label sets have to agree simultaneously.

## 4. What Is Not Covered

- No end-to-end HTTP test (supertest) for `RecommendationController` — consistent with the existing coverage gap in Goal/Roadmap/Assessment, not a new regression.
- No pagination on `GET /recommendation`.
- The "Overall Recommendation Score" is defined as an unweighted average of the other 5 scores — a defensible default, but one of several reasonable formulas; unlike `priorityScore` (whose weights are chosen to answer "what should be worked on next"), `overallScore`'s purpose is closer to "how much does this item matter overall," and that framing hasn't been validated against any real usage yet.
- As with Roadmap and Assessment, nothing in the codebase today automatically calls `POST /recommendation/generate` — there is still no live orchestration connecting Goal → Roadmap → Assessment → Recommendation into one pipeline. Each batch has delivered a correct, tested, isolated capability; wiring them together remains a distinct, not-yet-scheduled piece of work.

## 5. Classification

Per the batch's provided options (`NEEDS_REVISION`, `READY_FOR_LEARNING_SESSION_ENGINE`, `READY_FOR_DECISION_ENGINE`):

# READY_FOR_LEARNING_SESSION_ENGINE — Conditional

**Why not `NEEDS_REVISION`:** every domain-model element, every engine responsibility, every scoring-model value, every decision rule, all 8 learning strategies, all 4 commands/4 queries, all 7 API endpoints (plus history), all 5 metrics, and all 5 RBAC permissions in the spec are implemented and covered by a passing automated test; the full existing test suite (322 tests) has zero regressions; the engine cannot call an LLM, embedding service, vector database, or any external AI service even by accident — confirmed by inspecting every import in `domain/engine/`.

**Why not `READY_FOR_DECISION_ENGINE`:** this batch *is* the decision engine — classifying it as merely "ready for" a decision engine would be circular. The meaningful next question is what consumes this module's output next, and per the platform's own module names, that is a Learning Session engine (something that would take an approved `Recommendation` — its task priorities, review schedule, and strategy assignments — and turn it into an actual scheduled learning session for the learner). That is a well-bounded consumer: it can read `GetRecommendation`/`GetCompetencyProfile`-equivalent data without needing §3.2 (event semantics) or §3.3 (partial approval) resolved first, since a Learning Session engine would consume the *current, approved* recommendation state, not react to fine-grained change events or need item-level approval granularity on day one.

**Condition to move past Conditional:** confirm the intended semantics of `LearningStrategyChanged` (per-generation snapshot vs. cross-generation diff), decide whether item-level approval/rejection is actually required before a Learning Session engine is built against this contract, and — highest priority given how central this module is — establish the canonical `skillArea` source once, platform-wide (e.g., require it be a stable `RoadmapPhase.id` everywhere, not a free-text title), since three modules now depend on that string matching exactly.
