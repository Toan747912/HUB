# Canonical Constraint Catalog

## 1. Purpose

Define canonical standards for:
- Primary Key (PK)
- Foreign Key (FK)
- Unique constraints
- Check constraints
- constraint naming conventions

This is a governance catalog for WP-02 and does not generate SQL.

---

## 2. Constraint Naming Conventions

| Constraint Type | Pattern | Example |
|---|---|---|
| Primary Key | `pk_<table>` | `pk_assessment_result` |
| Foreign Key | `fk_<table>__<target_table>__<column>` | `fk_assessment_result__knowledge_node__knowledge_node_id` |
| Unique | `uq_<table>__<column_list>` | `uq_knowledge_node_mastery__learner_id__knowledge_node_id` |
| Check | `ck_<table>__<semantic_rule>` | `ck_decision_header__decision_type` |
| Exclusion/Partial Unique (if used) | `uqx_<table>__<semantic_scope>` | `uqx_learning_session__active_per_learner` |

Rules:
1. Names must be deterministic and globally unique within schema.
2. Do not use autogenerated anonymous names in canonical artifacts.
3. Constraint names should remain stable across migrations.

---

## 3. Primary Key Standards

1. Every canonical table must have exactly one PK.
2. PK column naming standard: `<entity>_id`.
3. PK must be immutable.
4. Bridge tables may use:
   - surrogate PK, or
   - deterministic composite PK (if explicitly governed).

PK policy by table class:
- Mutable snapshots: surrogate PK required.
- Append-only ledgers: surrogate PK required.
- Bridge mappings: surrogate PK preferred unless composite PK is clearly justified.

---

## 4. Foreign Key Standards

FKs must follow classifications from `Cross_Domain_FK_Strategy.md`:

- Physical FK
- Soft FK
- Projection-only
- Event-only dependency

### 4.1 Physical FK rules
1. Required for strict referential invariants.
2. Must specify delete/update action policy.
3. FK columns must have index coverage.

### 4.2 Soft reference rules
1. Must include validation strategy.
2. Must include ownership rationale.
3. Must not create implicit cross-domain write rights.

### 4.3 Cross-domain FK constraints
1. Cross-domain physical FK allowed only where boundary-safe.
2. Use event/projection strategies to avoid ownership inversion.
3. Recommendation and Teaching boundaries must remain protected.

---

## 5. Unique Constraint Standards

Unique constraints categories:

1. **Identity uniqueness**
   - entity natural keys or external IDs where required.

2. **Lifecycle uniqueness**
   - active singleton patterns (e.g., one active session per scope).

3. **Bridge uniqueness**
   - prevent duplicate relationship rows.

4. **Decision linkage uniqueness**
   - enforce one-to-one optional link patterns where decision headers are attached to specific artifacts.

Rules:
- Partial unique usage must be explicit and justified.
- Unique constraints should align with domain invariants, not convenience assumptions.

---

## 6. Check Constraint Standards

Check constraints required for:

1. Controlled enum-like domain values.
2. Non-empty semantic text fields (where rationale required).
3. Numeric bounds/ranges.
4. Lifecycle state validity patterns.

Examples of check policy classes:
- actor type validity
- decision type validity
- intervention tier validity
- self-reported level validity
- non-empty reasoning fields

Rules:
1. Check constraints must be readable and semantic.
2. Duplicated check logic should use standardized patterns.
3. Check policies must not conflict with event-driven transition workflows.

---

## 7. Constraint Catalog by Canonical Policy Area

## 7.1 Domain ownership safety constraints
- Ensure single write owner pathways cannot be violated by relational shortcuts.
- Enforce mapping uniqueness where ownership bridge exists.

## 7.2 Mastery and assessment constraints
- `knowledge_node_mastery` uniqueness by learner-node pair.
- assessment-to-knowledge referential integrity.
- Assessment remains sole owner of mastery mutation semantics.

## 7.3 Recommendation constraints
- Proposal artifact uniqueness and lifecycle consistency.
- No constraint pattern should imply recommendation ownership beyond proposal tables.

## 7.4 Session constraints
- Active session uniqueness patterns where applicable.
- Transition integrity constraints on sequence/lifecycle progression.

## 7.5 Decision/Explainability constraints
- decision type checks in decision header.
- detail linkage integrity to decision header.
- trace source/target type consistency checks.

---

## 8. Canonical Constraint Governance Checklist

Before a constraint is approved canonical:
1. Does it enforce a real invariant?
2. Does it preserve ownership boundaries?
3. Does it align with event-based orchestration rules?
4. Is it named by canonical convention?
5. Is index support defined if FK/lookup-critical?
6. Does it avoid conflicting with soft-reference/event-only models?

---

## 9. Forbidden Constraint Patterns

1. FK enforcing forbidden write coupling across domains.
2. Unique constraints that block valid append-only event ledgers.
3. Check constraints with ambiguous/non-deterministic semantics.
4. Inconsistent naming patterns within same table/domain class.
5. Constraints that encode business workflow transitions better handled by application/state machine contracts.
